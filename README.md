Пример многомодуьного приложения под андроид полностью на compose и с использованием сервислокатора и dagger2.
Часть кода надо причесать, но это скорее пример использования многомодульности, сервислокатора с использованием компонент-холдеров и
работы со слабыми и сильными ссылками на даггер-компоненты внутри них(и то же самое с зависимостями)

У каждого компонента (компонент на фичу) есть множество ключей фич, от которых он зависит и есть множество ключей компонентов,которые используют этот компонент(уникальный, обновляемый при переиспользовании фабрики компонента) айдишник).
При использовании компонента ему в множество владельцев добавляется ключ вызывающего его компонента, а при уходе с экрана происходит удаление ключа з множества(и так каскадно для каждой зависимости)

Структура приложения: есть app-модуль (он собирает в себе все остальные), core-модули (содержат базовые зависимости для фич и не зависят друг от друга), фиче-модули(фичи разделены как правило на публичную апи и закрытую импл часть, подключение одной фичи к другой осуществляется через апи-модуль). 
Навигация и вёрстка на чистом compose. Есть корневой граф для всего приложения, есть подграфы - один на фичу. Навигация производится по главному роуту фичи и дополнительно по роуту экрана внутри фичи. У каждой фичи есть вьюмодель, в которой производится подключение/удаление зависимостей.(Прокидывание этой вьюмодели выглядит слабо,это ещё надо доработать)
Фичи Main,Settings и Payments привязаны к табам на главном экране. В Фичу DFeature можно попасть из фичи Settings с её первого экрана

Как работает менеджер зависимостей вкратце:

Есть объект-синглтон FeatureServiceLocator(в модуле base-component-lifecycle),в котором хранится хешмапа factories с объектами-фабриками(с типом IFeatureComponentFactory).
Фабрика содержит в себе:
1) Ссылку на даггер-компонент для фичи( слабая ссылка на компонент, а сильная на слабую. Это нужно на тот случай, когда не надо всякий раз удалять компонент при выходе из фичи: можно просто присвоить null сильной ссылке и при повторном открытии фичи присвоить сильной ссылке слабую,если компонент не был уже удалён сборщиком мусора. Эта опция регулируется в методе фабрики removeComponent параметром hardRemove)б
2) Множество ключей фабрики тех компонентов(считай фич), от которых зависит текущая фича(при создании/удалении рекурсивно вызывается метод создания удаления для каждой фабрики из этого списка перед тем,как поработать с компонентом текущей фичи) - это dependenciesKeysList
3) Множество ключей тех компонентов, которые используют текущую фичу (которые зависят от текущей фичи) - это featureOwners. 
Каждый раз, когда фича Мейн например использует фичу Сеттингс, она добавляет свой уникальный ключ в множество featureOwners как своё, так и фичи Сеттингс(если сеттингс зависит от каких-то фич, то ключ компонента мейн будет добавлен для всех таких фич)
Каждый раз, когда например осуществляется выход из фичи Мейн, то у неё и у всех её зависимостей рекурсивно будет удалён уникальный ключ компонента Мейн из featureOwners. Компонент не будет удалён, пока его featureOwners не пуст.
Этот механизм позволяет эффекетивно управлять памятью в разных случаях:
 В случае повторного вызова одной и той же фичи(одного и того же экрана), при перекрёстных зависимостях(как в примере: мейн зависит от сеттингс,а сеттингс от мейн через прокси),
 В случае, когда фича использует зависимость другой фичи,экран которой ещё не открыт(экран Мейн зависит от Сеттингс, при этом экран Сеттингс не был открыт)
  и удалять ненужные зависимости при различных эджкейсах

Существует ключ для фибрики (FeatureKey) и ключ для компонента (экземпляр класса FeatureInstanceOwnerKey), который состоит из имени фичи и числа, число инкрементится всякий раз, когда вызывается конструктор у корневой вьюмодели фичи. 
Работа с ключами и их хранение осуществляется в объекте FeatureOwnerKeyRegistry.

Эджкейсы:
Сейчас в приложении есть пример перекрёстной зависимости, решённой через инверсию зависимостей( фича Мейн зависит от Сеттингс и наоборот)
В фиче Мейн: зависимости из Сеттингс подключаются через MainFeatureDependenciesModule(через provideSettingsFeatureDependencies тянется компонент фичи Сеттингс, и с помощью метода provideGetSettingsDetailLevelUseCase берётся нужный объект)
В фиче Сеттингс: зависимости из Мейн грузятся в фабрике Сеттингс в методе getComponent (val mainDependencies =
featureServiceLocator.getFactory(FeatureKey.MAIN)
.getComponent(ownerKey = ownerKey) as MainFeatureDependencies) и прокидываются в билдер. При присваивани значения mainDependencies заодно добавляется ключ featureOwner, это СТРОГО необходимо делать перед возвратом сильной ссылки на компонент
